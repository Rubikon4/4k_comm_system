# Уведомления

Этот документ описывает события, формирующие уведомления, и правила доставки.
Структура модели `Notification` — в [docs/data_model.md](data_model.md).
Создание уведомлений идёт исключительно через `apps/notifications/services.py`,
который вызывается из service layer других app по бизнес-событиям.

## 1. События, формирующие уведомления

### По задачам

- **`task_assigned`** — пользователь назначен исполнителем задачи; также отправляется
  при создании клона повторяющейся задачи — каждый исполнитель клона получает уведомление
  (иначе исполнитель не узнает, что цикл снова запущен).
- **`task_status_changed`** — изменился статус задачи, в которой пользователь — исполнитель
  или постановщик.
- **`task_sent_to_review`** — задача отправлена на уточнение (отдельное событие, более
  заметное, чем общее `task_status_changed`).
- **`task_worker_done`** — исполнитель завершил задачу (уведомление получает постановщик).
- **`task_head_done`** — руководитель завершил задачу (уведомление получает исполнитель).

### По рабочим группам

- **`workgroup_added`** — пользователь добавлен в рабочую группу.

### По чатам

- **`chat_added`** — пользователь добавлен в чат.
- **`chat_new_message`** — в чате появилось новое сообщение
  (с **троттлингом**, см. раздел 3).

### По файлам

- **`attachment_added`** — добавлен файл, связанный с доступной пользователю задачей или
  группой. Уведомление НЕ дублируется, если пользователь и так получит уведомление
  о действии (например, "файл прикреплён к задаче при её создании" не порождает
  отдельного `attachment_added`).

> **Статус реализации:** тип события `attachment_added` определён в `Notification.EventType`,
> но функция `notify_attachment_added` в `apps/notifications/services.py` **не реализована**
> (отложена как направление развития). История прикрепления файла к задаче записывается
> через `TaskHistory` (`attachment_added` / `attachment_removed`), но уведомление другим
> участникам не отправляется.

## 2. Получатели уведомлений

Для каждого события service layer определяет получателей по правилам:

| Событие | Получатели |
|---------|------------|
| `task_assigned` | назначенный исполнитель (в т.ч. исполнители клона повторяющейся задачи) |
| `task_status_changed` | все активные исполнители + постановщик (кроме того, кто сам выполнил действие) |
| `task_sent_to_review` | все активные исполнители + постановщик (кроме инициатора) |
| `task_worker_done` | постановщик задачи |
| `task_head_done` | все активные исполнители |
| `workgroup_added` | добавленный пользователь |
| `chat_added` | добавленный пользователь |
| `chat_new_message` | все активные участники чата, кроме автора сообщения, с учётом троттлинга |
| `attachment_added` | пользователи, имеющие доступ к связанному объекту, кроме автора загрузки |

Никогда не уведомлять пользователя о действии, которое он выполнил сам.

## 3. Троттлинг уведомлений о новых сообщениях в чате

Это самое нетривиальное правило в системе. Сформулировано чётко, чтобы не путать
реализацию.

### Правило

Для события `chat_new_message`:

> Если у пользователя по этому чату уже есть **непрочитанное** уведомление о новых сообщениях,
> новое уведомление **не создаётся**.
> Если непрочитанного уведомления о новых сообщениях по этому чату нет — уведомление
> создаётся.

То есть: на каждый чат у каждого пользователя в любой момент времени может быть **не более
одного непрочитанного** уведомления о новых сообщениях. По другим чатам уведомления
создаются независимо.

### Когда счётчик "сбрасывается" (становится возможным новое уведомление)

Уведомление перестаёт блокировать создание следующего, когда:

1. **пользователь прочитал** это уведомление (нажал на него или отметил как прочитанное)
   → `is_read = True`, `read_at = now()`;
2. **пользователь зашёл в чат** (открыл страницу чата) → все непрочитанные уведомления
   `chat_new_message` по этому чату для этого пользователя помечаются прочитанными.

После любого из этих событий следующее новое сообщение в чате снова создаст уведомление.

### Алгоритм в service layer

```
def notify_chat_new_message(chat, message, recipients):
    for user in recipients:
        existing = Notification.objects.filter(
            recipient=user,
            event_type='chat_new_message',
            object_type='chat',
            object_id=chat.id,
            is_read=False,
        ).exists()
        if existing:
            continue  # троттлинг: не создаём дубль
        Notification.objects.create(
            recipient=user,
            event_type='chat_new_message',
            object_type='chat',
            object_id=chat.id,
            text=f'Новые сообщения в чате "{chat.name}"',
        )
```

### Открытие чата сбрасывает все его уведомления

При открытии страницы чата view выполняет:

```
Notification.objects.filter(
    recipient=request.user,
    event_type='chat_new_message',
    object_type='chat',
    object_id=chat.id,
    is_read=False,
).update(is_read=True, read_at=timezone.now())
```

Это сбрасывает счётчик и одновременно обновляет `ChatMembership.last_seen_at`.

## 4. Что хранится в уведомлении

Поля модели — в [docs/data_model.md](data_model.md), раздел 5.

Связь со связанным объектом — через простые поля `object_type` и `object_id`,
без `GenericForeignKey`. Текст уведомления формируется в момент создания и сохраняется
как есть; если связанный объект потом удалится или переименуется, текст уведомления
останется историческим, что для прототипа допустимо.

## 5. Отображение уведомлений

- На dashboard — счётчик непрочитанных уведомлений в шапке.
- Список уведомлений — отдельная страница `/notifications/`.
- Каждое уведомление имеет ссылку на связанный объект (формируется по `object_type` и
  `object_id`).
- Клик по уведомлению — переход на связанный объект + пометка как прочитанное.
- Группового сброса "пометить всё как прочитанное" в MVP **нет** (можно добавить позже).

## 6. Чего нет в первой версии

- email-уведомлений;
- push-уведомлений в браузере;
- desktop/mobile push;
- WebSocket-доставки уведомлений в реальном времени (счётчик в шапке
  обновляется через `context_processor` при каждом переходе между страницами).
