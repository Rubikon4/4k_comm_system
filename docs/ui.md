# Интерфейс и UX

Этот документ описывает страницы приложения и принципы UX. Технические аспекты
(стек frontend, polling) — в [docs/architecture.md](architecture.md).

## 1. Общие принципы

- **Язык интерфейса** — русский.
- **Стек** — Django Templates + Bootstrap 5 + минимальный vanilla JavaScript.
- **Цель** — продемонстрировать рабочие сценарии, а не идеальный дизайн.
- **Принцип "одного контекста"**: окно задачи и окно рабочей группы НЕ являются
  отдельными страницами. Они открываются как модальные окна (Bootstrap modal) или
  выдвижные панели поверх списка, чтобы пользователь не терял контекст списка.

## 2. Список страниц (полных, с собственным URL)

| URL | Назначение |
|-----|------------|
| `/login/` | страница входа (стандартная Django auth) |
| `/logout/` | выход (POST) |
| `/` | dashboard (персональное рабочее пространство) |
| `/tasks/` | список задач с фильтрами |
| `/workgroups/` | список рабочих групп |
| `/chats/` | список чатов |
| `/chats/<id>/` | страница конкретного чата |
| `/notifications/` | список уведомлений |
| `/profile/` | страница профиля пользователя |
| `/admin/` | Django Admin |

## 3. Окна, открывающиеся в модалках (не отдельные страницы)

Для следующих сценариев — модальное окно поверх соответствующего списка:

- **Окно задачи** (детали + история + статус + вложения) — над списком задач.
- **Форма создания/редактирования задачи** — над списком задач или окном группы.
- **История изменений задачи** — внутри окна задачи (вкладка/раздел).
- **Окно рабочей группы** (детали + участники + дочерние группы + связанные задачи) —
  над списком групп.
- **Форма создания/редактирования группы** — над списком групп.
- **Форма добавления участника в группу** — над окном группы.

Это даёт ощущение единого рабочего пространства и упрощает возврат к списку.

## 4. Dashboard (`/`)

Главная страница после входа. Показывает срез "что у меня сейчас":

### Блоки dashboard

1. **Шапка** — навигация, профиль, **счётчик непрочитанных уведомлений** (popup-индикатор
   рядом с иконкой).
2. **Мои задачи** (открытый блок по умолчанию) — два переключаемых вида:
   «Назначены мне» (я исполнитель) и «Назначил я» (я постановщик);
   фильтрация по статусу и приоритету; ссылка на полный список.
3. **Мои рабочие группы** — список групп, в которых я активный участник; ссылка на
   полный список.
4. **Мои чаты** — список чатов, где я активный участник; индикатор новых сообщений
   (бэйдж по чатам с непрочитанными уведомлениями).
5. **Последние уведомления** — короткий список, ссылка на полный.

Dashboard важен для защиты ВКР: показывает, что система — не набор отдельных страниц,
а связанное рабочее пространство.

## 5. Список задач (`/tasks/`)

### Основной экран

- таблица или список задач;
- переключатель: «Назначены мне» / «Назначил я»;
- фильтры:
  - по статусу (`new`/`inprogress`/`review`/`workerdone`/`headdone`/`cancel`);
  - по приоритету (`low`/`normal`/`high`/`urgent`);
  - по постановщику (только в виде «Назначены мне»);
- сортировка по сроку (`deadline_date`) и дате создания;
- кнопка "Создать задачу" (видна только тем, кто имеет право создавать задачи).

### Окно задачи (модальное)

- название, описание, постановщик, исполнители;
- срок, приоритет, статус;
- кнопки смены статуса (видны согласно [docs/permissions.md](permissions.md));
- список вложений с кнопкой "Скачать";
- история изменений (вкладка или нижний раздел).

## 6. Список рабочих групп (`/workgroups/`)

### Основной экран

- иерархический список: родительские группы и под ними дочерние;
- фильтр "только мои" / "все доступные";
- кнопка "Создать родительскую группу" (только admin);
- кнопка "Создать дочернюю группу" внутри родительской (admin или parent_head).

### Окно группы (модальное)

- название, описание, тип (родительская/дочерняя);
- список участников с локальными ролями;
- список дочерних групп (если родительская);
- список задач группы (короткий, с переходом в полный список с фильтром);
- кнопка "Добавить участника" (по правам);
- ссылка на основной чат группы.

## 7. Список чатов (`/chats/`)

- список всех чатов, в которых пользователь — активный участник;
- группировка по типу: личные (`direct`), основные групповые (`workgroup`),
  дополнительные (`custom`);
- индикатор непрочитанных сообщений (бэйдж);
- кнопка "Создать личный чат" (любой пользователь);
- кнопка "Создать дополнительный чат" (только headworker / admin, согласно
  [docs/permissions.md](permissions.md), раздел 6).

## 8. Страница чата (`/chats/<id>/`)

- шапка: название чата, тип, участники (компактно);
- лента сообщений (хронологически, новые внизу);
- форма отправки сообщения с поддержкой вложений (текст + файл/изображение);
- автообновление через polling каждые 5 секунд (см.
  [docs/architecture.md](architecture.md), раздел 4);
- при открытии страницы — все непрочитанные уведомления `chat_new_message` по этому
  чату помечаются прочитанными (см. [docs/notifications.md](notifications.md), раздел 3).

## 9. Список уведомлений (`/notifications/`)

- хронологический список;
- разделение по статусу: непрочитанные / прочитанные;
- клик на уведомление → переход на связанный объект (задача, группа, чат) + пометка
  прочитанным;
- в MVP — без массового "отметить всё прочитанным".

## 10. Профиль пользователя (`/profile/`)

- имя, должность, телефон, аватар;
- редактирование собственных данных;
- смена пароля (стандартный Django flow);
- системная роль — отображается, но не редактируется (только admin может менять через
  Django Admin).

## 11. Административная панель (`/admin/`)

Стандартный Django Admin с настроенными `ModelAdmin` для:
- User и Profile (управление пользователями, ролями, блокировка);
- WorkGroup, WorkGroupMembership;
- Task, TaskAssignee, TaskHistory;
- Chat, ChatMembership, Message;
- Notification;
- Attachment.

Для простых аварийных вмешательств. Не заменяет основной интерфейс.

## 12. Минимальный JavaScript

Допустимый объём vanilla JavaScript:

- polling сообщений в чате (`fetch` каждые 5 секунд);
- обновление счётчика уведомлений (либо вместе с polling чата, либо при навигации);
- открытие/закрытие модальных окон (Bootstrap нативно умеет);
- AJAX-загрузка содержимого модальных окон, чтобы не дублировать рендер на сервере;
- подтверждение удаления (Bootstrap modal вместо `confirm()`).

JavaScript структурируется по файлам:
- `static/js/chat.js` — polling и работа с чатом;
- `static/js/notifications.js` — счётчик уведомлений;
- `static/js/modals.js` — общая загрузка модалок;
- `static/js/main.js` — общие хелперы.

Никаких сборщиков (Webpack, Vite). Просто статические файлы.

## 13. Чего нет в MVP

- календаря задач;
- сложной аналитики;
- диаграмм Ганта;
- глобального поиска по всему контенту;
- темы оформления (только светлая, дефолтная Bootstrap);
- мобильной адаптации сверх дефолтной адаптивности Bootstrap.
