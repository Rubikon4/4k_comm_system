# Права доступа

Этот документ описывает роли в системе и матрицу прав. Все проверки прав реализуются
**в service layer** (`apps/<module>/services.py`).

**Принцип**: знание о праве хранится в одном месте — в `apps/<module>/permissions.py`.
Применяется на нескольких рубежах: в service layer (обязательная жёсткая проверка),
в шаблонах (скрытие кнопок для UX). Это не дублирование, а **defence in depth**: каждый
рубеж защищает от своего класса угроз. Service layer — единственный рубеж, который
гарантированно срабатывает при любом способе вызова (HTTP, Django Admin, management-команда,
тест).

## 1. Два уровня ролей

В системе различаются **системные роли** (глобальные) и **локальные роли в группе**.

### Системные роли (поле `Profile.role`)

Глобальная роль пользователя в системе. У каждого — ровно одна.

- **`worker`** — сотрудник: выполняет задачи, общается в доступных чатах.
- **`headworker`** — руководитель: создаёт дочерние группы и задачи; системная роль
  будет расширяться по мере развития проекта.
- **`admin`** — администратор: полный доступ ко всем объектам системы.

### Локальные роли в группе (поле `WorkGroupMembership.local_role`)

Существуют **только в дочерних группах** (уровень 2 и глубже).
В родительских группах (уровень 1) локальных ролей нет — только системные.

- **`member`** — обычный участник.
- **`parent_head`** — руководитель группы; автоматически присваивается создателю при
  создании дочерней группы.
- **`child_head`** — назначаемый руководитель; назначается `parent_head` при создании
  группы или позже; необязателен; может быть несколько; **любой пользователь** независимо
  от системной роли.

### Ключевое различение

- `headworker` (системная) — даёт право создавать дочерние группы внутри родительских.
- `parent_head` / `child_head` (локальные) — определяют права внутри конкретной ветки
  дерева групп.
- В тексте и коде всегда указывать: «системная роль `headworker`» или «локальная роль
  `parent_head`» — не путать.

## 2. Иерархия групп и права на создание

Группы образуют дерево произвольной глубины. Ограничение «только два уровня» снято.

| Уровень | Тип | Кто создаёт | Локальные роли |
|---------|-----|-------------|----------------|
| 1 (корень) | Родительская | только `admin` | **нет** (только системные) |
| 2 | Первая дочерняя | `admin` или `headworker` в родительской | есть |
| 3+ | Вложенные | `admin`, `parent_head` или `child_head` группы-родителя | есть |

Создатель дочерней группы автоматически получает `parent_head` в ней.
`child_head` назначается `parent_head` при создании или позже. Назначение необязательно.
Может быть несколько `child_head` в одной группе.

**Особый случай**: `admin` создаёт группу с участниками из разных родительских групп →
она считается новой родительской (корень дерева, `parent = null`).

### Деактивация групп

Физического удаления групп нет (только через Django Admin как аварийное действие).

При деактивации группы (`is_active = False`):
- рекурсивно деактивируются все дочерние группы и подгруппы;
- членства в этих группах деактивируются (`WorkGroupMembership.is_active = False`);
- задачи постановщиков и исполнителей из этих групп **остаются нетронутыми**;
- реактивация — только `admin` вручную.

**Видимость деактивированных групп:**

- неактивные группы **скрываются** из списков и интерфейсов всех пользователей
  (включая `parent_head`, `child_head` и `headworker`);
- единственное место, где они видны — Django Admin (`/admin/`), доступный только
  `admin`;
- восстановление (реактивация) — исключительная прерогатива `admin` через
  Django Admin;
- любая фильтрация в queryset на уровне views и services должна включать
  `is_active=True` для группы (и её предков, если идёт обход дерева).

Цель — не засорять рабочие списки группами, которые «удалили» (мягко), но
сохранять данные для аудита и возможности восстановления.

## 3. Управление членством в группах

| Уровень группы | Добавить участника | Изменить роль | Деактивировать участника |
|----------------|-------------------|---------------|--------------------------|
| Родительская (уровень 1) | только `admin` | только `admin` | только `admin` |
| Дочерняя (уровень 2+) | `admin` или `parent_head` этой группы | `admin` или `parent_head` этой группы | `admin` или `parent_head` этой группы |

## 4. Создание задач и назначение исполнителей

Задача **не привязана к группе** (нет поля `Task.workgroup`). Принадлежность задачи
определяется её постановщиком и исполнителями.

### Кто может создать задачу и кому назначить

Создатель задачи может назначить её только пользователям из своего **поддерева групп**
(своя группа + все вложенные ниже):

| Кто создаёт задачу | Кому может назначить |
|--------------------|---------------------|
| `admin` | любому пользователю системы |
| `headworker` в родительской группе | участникам этой родительской + всех её дочерних |
| `parent_head` дочерней группы | участникам этой дочерней + всех её вложенных |
| `child_head` (он же `parent_head` в подгруппе) | участникам группы, где он `parent_head`, + всех её вложенных |
| `worker` | только самому себе **или** другому `worker`'у в **той же** дочерней группе |

`worker` — единственная роль, которой ограничено назначение: он не может назначить
задачу вверх по иерархии или в другую группу.

### Видимость задачи

Конкретную задачу могут **видеть** (открыть окно задачи):
- постановщик (`Task.created_by`);
- активные исполнители (`TaskAssignee.is_active = True`);
- `admin`.

Участники группы, не являющиеся постановщиком или исполнителем, задачу **не видят**.
Принадлежность задачи к группе в явном виде не хранится — фильтрация по правам
выполняется через `created_by` и `TaskAssignee`.

### Изменение задачи

| Действие | Кто может |
|----------|-----------|
| Изменить срок / приоритет / описание | постановщик или `admin` |
| Добавить / удалить исполнителя | постановщик или `admin` |

## 5. Жизненный цикл задачи: кто меняет статус

`admin` может вмешаться на любом этапе и выполнить любой переход.

| Переход | Кто может (кроме `admin`) |
|---------|---------------------------|
| `* → new` (создание) | постановщик |
| `new → inprogress` | назначенный исполнитель |
| `inprogress → review` | назначенный исполнитель |
| `review → inprogress` | назначенный исполнитель |
| `inprogress → workerdone` | назначенный исполнитель |
| `workerdone → review` | постановщик |
| `workerdone → headdone` | постановщик |
| `new / inprogress / review / workerdone → cancel` | постановщик |

Исполнитель НЕ может перевести задачу в `headdone` — это право постановщика.

**Запрещённые переходы (не реализовывать):**
- `headdone → cancel` — нельзя. Задача уже завершена постановщиком; отмена после
  завершения — переписывание истории.
- `cancel → *` — нельзя. Отмена необратима; для возобновления работы создаётся
  новая задача вручную.

## 6. Чаты: создание и управление участниками

### Типы чатов

- **`direct`** — личный чат 1:1.
- **`workgroup`** — основной чат рабочей группы; создаётся автоматически при создании
  WorkGroup; состав синхронизируется с WorkGroupMembership.
- **`custom`** — произвольный групповой чат, не привязанный к WorkGroup; создаётся
  вручную.

### Право создания `direct`-чата (1:1)

| Системная роль | С кем может создать |
|----------------|---------------------|
| `worker` | только с пользователем, с которым состоит в одной общей группе (любого уровня) |
| `headworker` | с любым пользователем системы |
| `admin` | с любым пользователем системы |

### Право создания `custom`-чата

| Системная роль | Кого может включить |
|----------------|---------------------|
| `worker` | только пользователей, с которыми состоит в одной общей группе (любого уровня) |
| `headworker` | любых пользователей системы |
| `admin` | любых пользователей системы |

Подтверждения `admin` для любого типа чата **не требуется**.

### `workgroup`-чат

Создаётся автоматически при создании WorkGroup. Ручное добавление/удаление участников
невозможно — состав управляется исключительно через WorkGroupMembership.

**Синхронизация состава:** изменения в `WorkGroupMembership` (добавление/деактивация
участника) автоматически отражаются в составе `workgroup`-чата. Реализуется через
Django **signal** (`post_save` на `WorkGroupMembership`) в `apps/workgroups/signals.py`.
Сигнал вызывает `apps/chats/services.py::sync_workgroup_chat_members(workgroup)`.

### Защита direct-чата

При создании direct-чата в `apps/chats/services.py` service layer обязан:

1. Проверить, что `user1 != user2` (нельзя создать чат с самим собой).
2. Проверить, нет ли уже активного direct-чата между этой парой пользователей
   (`Chat.objects.filter(chat_type='direct').filter(chatmembership__user=user1).filter(chatmembership__user=user2).first()`).
   - Если чат найден — вернуть существующий объект вместо создания нового.
   - Если нет — создать новый.

С точки зрения UI пользователь просто оказывается в чате — без сообщения об ошибке.

### Управление режимом отправки сообщений

Создатель чата или `admin` могут:

- **Закрыть чат на запись** (`Chat.is_writable = False`) — все участники переходят в
  режим «только чтение»; создатель и `admin` могут снова открыть.
- **Заглушить конкретного участника** (`ChatMembership.can_write = False`) — участник
  читает, но не отправляет.

При отправке сообщения service layer проверяет оба условия:
`chat.is_writable == True` И `membership.can_write == True`.

## 7. Чтение и отправка сообщений

| Действие | Кто может |
|----------|-----------|
| Читать сообщения | активный участник чата (`ChatMembership.is_active = True`) |
| Отправлять сообщения | активный участник при `is_writable = True` и `can_write = True` |
| Редактировать своё сообщение | автор (временное окно — обсуждается отдельно) |
| Мягко удалить своё сообщение | автор или `admin` |
| Закрыть/открыть чат на запись | создатель чата или `admin` |
| Заглушить/разглушить участника | создатель чата или `admin` |

**Поведение при `ChatMembership.is_active = False`:**
Пользователь **не видит чат** в списке чатов и не может зайти на его страницу.
Сообщения в чате остаются в БД для аудита, но пользователю недоступны. View
фильтрует чаты через `.filter(chatmembership__user=request.user, chatmembership__is_active=True)`.
Бывший участник не может скачать файлы, прикреплённые к сообщениям этого чата.

**Мягкое удаление (soft delete)**: `Message.is_deleted = True`. Физически запись остаётся
в БД. В UI отображается как «Сообщение удалено» (без исходного текста). Данные сохраняются
для аудита и потому что вложения (`Attachment`) ссылаются на сообщение через FK.

## 8. Файлы и вложения

Файлы крепятся **только** к `Task`, `Message` или `WorkGroup`.

| Действие | Кто может |
|----------|-----------|
| Прикрепить файл к задаче | постановщик, активный исполнитель или `admin` |
| Прикрепить файл к сообщению | автор сообщения (в момент отправки) |
| Прикрепить файл к группе | `parent_head` этой группы или `admin` |
| Скачать файл задачи | постановщик, активный исполнитель или `admin` |
| Скачать файл сообщения | активный участник этого чата или `admin` |
| Скачать файл группы | активный участник этой группы или `admin` |
| Удалить вложение | загрузивший пользователь или `admin` |

Скачивание реализуется только через защищённый view с проверкой прав
(см. [docs/architecture.md](architecture.md), раздел 3).

## 9. Уведомления

| Действие | Кто может |
|----------|-----------|
| Видеть уведомления | только получатель (`Notification.recipient = request.user`) |
| Помечать как прочитанные | только получатель |
| Создавать уведомления | системно из service layer; вручную не создаются |

## 10. Администрирование

`admin` имеет доступ ко всем объектам системы и может:
- создавать и блокировать пользователей;
- назначать системные роли;
- создавать родительские группы;
- деактивировать группы (каскадно);
- вмешиваться в жизненный цикл задач на любом этапе;
- использовать Django Admin для аварийных действий.

## 11. Где реализуется проверка прав

| Слой | Роль | Обязателен |
|------|------|-----------|
| `apps/<module>/permissions.py` | единое место знания о праве | да |
| Service layer (`services.py`) | жёсткая проверка с `PermissionDenied` | **да** |
| Views (`views.py`) | `@login_required`, `is_active` | да |
| Шаблоны (`.html`) | скрытие кнопок для UX | желательно, не обязательно |
| Django Admin (`admin.py`) | ограничение видимости в /admin/ | где критично |

Обязательный и единственный надёжный рубеж — **service layer**.
Остальные слои — для удобства и UX, но не как замена проверке в сервисе.
