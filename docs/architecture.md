# Архитектура проекта

Документ фиксирует архитектурные решения для внутренней ИС компании "4К".
Все решения приняты с учётом формата прототипа для защиты ВКР.

## 1. Технологический стек (зафиксирован)

| Слой | Решение |
|------|---------|
| Язык | Python 3.12+ |
| Фреймворк | Django 5.x (LTS) |
| База данных | PostgreSQL 16 |
| Сервер приложения | Gunicorn (WSGI), 4 worker'а по умолчанию |
| Контейнеризация | Docker Compose (только два сервиса: `web` и `db`) |
| Frontend | Django Templates + Bootstrap 5 + vanilla JavaScript |
| Раздача статики | WhiteNoise (через web-контейнер, без отдельного nginx) |
| Аутентификация | Стандартная Django (сессии, login_required) |

**TypeScript не используется.** Только vanilla JavaScript для минимальных интерактивных элементов
(polling в чате, обновление счётчиков уведомлений, AJAX-загрузка модальных окон).

### Запрещено без отдельного обоснования и подтверждения

React, Vue, отдельный SPA frontend, Django REST Framework, WebSocket, Django Channels, Redis,
Celery, микросервисы, Kubernetes, Kafka, ASGI-сервер, дополнительные контейнеры
(кроме `web` и `db`).

## 2. Контейнеризация

Разворачивается через `docker compose up`.

### Сервисы

- **`web`** — Django-приложение под управлением Gunicorn.
- **`db`** — PostgreSQL 16.

### Volumes (тома Docker)

**Volume** — это область диска, которую Docker подключает к контейнеру. Данные в volume
переживают пересборку и пересоздание контейнера.

| Volume | Назначение | Куда монтируется |
|--------|------------|------------------|
| `postgres_data` | данные PostgreSQL | в `db`, путь `/var/lib/postgresql/data` |
| `media_data` | пользовательские файлы (`MEDIA_ROOT`) | в `web`, путь `/app/media` |
| `static_data` | собранная статика (`STATIC_ROOT`) | в `web`, путь `/app/staticfiles` |

## 3. Хранение файлов и вложений

**Решение: файлы хранятся через Docker volume `media_data`, подключённый к web-контейнеру.**

Обоснование:
- хранить внутри образа web нельзя — пересборка удалит все данные;
- хранить в PostgreSQL нельзя — раздувание БД, медленные `pg_dump`, антипаттерн;
- сторонние объектные хранилища (S3, MinIO) исключены ограничением "только два контейнера".

Django настраивается:
- `MEDIA_ROOT = /app/media` (внутри web-контейнера);
- `MEDIA_URL = /media/` (только для внутренних ссылок, **не для прямого скачивания**);
- volume `media_data` монтируется в `/app/media`.

### Безопасность доступа к файлам

Прямая отдача файлов через `MEDIA_URL` запрещена для авторизованных вложений.
Скачивание идёт через специальный view `/attachments/<id>/download/`, который:

1. проверяет авторизацию пользователя;
2. проверяет, имеет ли пользователь доступ к связанному объекту (Task, Message, WorkGroup);
3. отдаёт файл через `FileResponse` с заголовком
   `Content-Disposition: attachment; filename="<original_name>"` — файл скачивается,
   а не открывается в браузере.

Ограничения при загрузке:
- максимальный размер: **35 МБ** (`MAX_UPLOAD_SIZE` в settings);
- разрешённые расширения (`ALLOWED_UPLOAD_EXTENSIONS` в settings):
  `pdf`, `doc`, `docx`, `xls`, `xlsx`, `ppt`, `pptx`, `txt`, `csv`,
  `jpg`, `jpeg`, `png`, `gif`, `zip`, `rar`, `7z`;
- MIME-тип определяется через библиотеку **`python-magic`** (`requirements.txt`);
  не доверяем `Content-Type` из браузера — он подделывается.

## 4. Чат: формат обмена сообщениями

**Используется polling.** Это термин для повторяющихся запросов клиента к серверу
с заданным интервалом. Клиент сам спрашивает "есть ли что-то новое?", сервер отвечает
короткими порциями данных.

### Параметры

- интервал по умолчанию: **5 секунд**;
- если вкладка свёрнута (`document.hidden`): интервал увеличивается до 30 секунд или
  останавливается до возвращения фокуса;
- запрос НЕ перезагружает страницу полностью.

### Реализация

Endpoint вида `GET /chats/<id>/messages/?since=<last_message_id>` возвращает только
новые сообщения после указанного `id`. JavaScript на странице ведёт `lastMessageId`,
делает `fetch`, добавляет новые сообщения в DOM.

### Почему не WebSocket

WebSocket требует ASGI-сервер, Django Channels и Redis (channel layer).
Это противоречит ограничениям стека и не оправдано для 47–70 пользователей.

При нагрузке 70 одновременно активных чатов: 70 запросов / 5 секунд = 14 RPS.
Это пренебрежимо мало для Gunicorn + Django + PostgreSQL.

Граница, после которой WebSocket становится оправданным:
- 200+ одновременно активных пользователей в чатах;
- требование задержки доставки <1 секунды;
- индикаторы "печатает...", "прочитано в реальном времени".

В рамках ВКР переход на WebSocket указывается как направление развития.

## 5. Service layer (слой бизнес-логики)

**Service layer** — это набор функций или классов, которые выполняют бизнес-операции
(создать задачу, сменить статус, добавить участника). Они вызываются из views, Django Admin,
тестов и management-команд.

### Расположение

Файлы `services.py` создаются **внутри каждого app**, а не в `core`:

- `apps/tasks/services.py` — операции над задачами (включая создание клона при `headdone` для повторяющихся задач);
- `apps/workgroups/services.py` — операции над группами и членством;
- `apps/chats/services.py` — операции над чатами и сообщениями, в т.ч. `sync_workgroup_chat_members(workgroup)`;
- `apps/notifications/services.py` — создание уведомлений с учётом троттлинга;
- `apps/attachments/services.py` — загрузка (валидация расширения + MIME через python-magic) и проверка прав на скачивание.

### Синхронизация состава workgroup-чата

Когда пользователь добавляется или деактивируется в `WorkGroupMembership`, нужно
автоматически обновить участников `workgroup`-чата этой группы.

Реализация: Django **сигнал** (`post_save` на `WorkGroupMembership`) в файле
`apps/workgroups/signals.py`. Сигнал вызывает
`apps/chats/services.py::sync_workgroup_chat_members(workgroup)`, которая:
- добавляет нового участника в `ChatMembership` (если ещё не там);
- деактивирует запись `ChatMembership` при деактивации `WorkGroupMembership`.

Сигналы подключаются в `apps/workgroups/apps.py` через `AppConfig.ready()`.

В `apps/core/` остаются только:
- общие миксины (`TimestampedModel`, миксины проверки прав);
- утилиты (валидаторы файлов, хелперы);
- базовые шаблоны.

### Зачем это нужно

- **Тонкие views**: view получает запрос, проверяет форму, вызывает сервис.
- **Единое место для бизнес-правил**: проверки прав, запись истории, создание уведомлений
  происходят в одном месте, а не дублируются между views и Django Admin.
- **Тестируемость**: бизнес-логика тестируется без HTTP, без request-объектов.
- **Прозрачность для защиты ВКР**: чёткое разделение слоёв — это аргумент в защите.

## 6. User и Profile

**Решение: используется стандартный `django.contrib.auth.models.User` + отдельная модель
`Profile` через `OneToOneField`.**

### Объяснение разницы между "полем" и "моделью"

**Поле в существующей модели** — это столбец в таблице. Чтобы добавить поле в стандартный
User, нужно создать кастомного User (`AbstractUser` или `AbstractBaseUser`). Это решение
**нельзя поменять задним числом без болезненной миграции всей схемы БД**.
Поэтому кастомного User делают только тогда, когда заранее точно известно, что он нужен.

**Отдельная модель Profile** — это вторая таблица, связанная с User один-к-одному.
Расширяет данные пользователя без изменения стандартной таблицы аутентификации Django.

### Преимущества Profile

1. **Не ломает стандартную аутентификацию.** Работают все встроенные views Django (`login`,
   `logout`, `password_reset`), Django Admin, готовые формы.
2. **Не требует кастомного UserManager.** Меньше кода, меньше шансов ошибиться.
3. **Гибкость без миграций User.** Поля Profile можно добавлять и удалять отдельно.
4. **Чёткое разделение ответственности**: User отвечает за аутентификацию (логин, пароль,
   права Django), Profile — за бизнес-данные (должность, телефон, аватар).
5. **Безопасный путь для прототипа**: если позже понадобится кастомный User, миграция
   с `User + Profile` на `CustomUser` всё ещё возможна; обратный путь — нет.

### Что хранится где

- `User` (стандартный): username, email, first_name, last_name, password, is_active,
  is_staff, is_superuser, date_joined.
- `Profile`: должность, телефон, аватар, системная роль (worker/headworker/admin),
  возможные бизнес-флаги.

Подробности полей — в [docs/data_model.md](data_model.md).

## 7. Производительность и индексы

Прототип рассчитан на 47–70 пользователей, но индексы должны быть заложены сразу,
чтобы не переписывать запросы потом.

### Обязательные индексы

Полный список индексов — в [docs/data_model.md](data_model.md), раздел 8.
Источник истины для индексов один — `data_model.md`. Здесь, в архитектуре,
описано только обоснование подхода.

### Обязательные оптимизации queryset'ов

- `select_related('author')` для списка сообщений чата;
- `select_related('author', 'group')` для списка задач;
- `prefetch_related('assignees')` для списка задач с исполнителями;
- `LIMIT` на endpoint polling-чата (например, 50 сообщений за один ответ).

## 8. Развёртывание

### Файлы конфигурации

- `Dockerfile` — образ web (Python + Django + Gunicorn + WhiteNoise);
- `docker-compose.yml` — сервисы web и db, volumes, network;
- `.env.example` — шаблон переменных окружения;
- `.env` — реальные значения (в `.gitignore`);
- `requirements.txt` — зависимости Python с пинами версий;
- `manage.py` — стандартный Django.

### Структура settings

- `config/settings/base.py` — общие настройки;
- `config/settings/dev.py` — DEBUG=True, локальная БД, расширенный логинг;
- `config/settings/prod.py` — DEBUG=False, ALLOWED_HOSTS, security headers, WhiteNoise.

Выбор окружения — через `DJANGO_SETTINGS_MODULE` в `.env`.

## 9. Структура Django apps

```
apps/
├── core/           # миксины, утилиты, базовые шаблоны, проверки прав
├── accounts/       # User (стандартный), Profile, авторизация, системные роли
├── workgroups/     # WorkGroup, WorkGroupMembership
├── tasks/          # Task, TaskAssignee, TaskHistory
├── chats/          # Chat, ChatMembership, Message
├── notifications/  # Notification
├── attachments/    # Attachment
└── dashboard/      # персональное рабочее пространство (только views/templates)
```

`notifications` и `attachments` — отдельные apps (не сливаются с другими модулями).
Это даёт чёткие границы ответственности и упрощает демонстрацию архитектуры на защите.

## 10. Связи между модулями (направление импортов)

Чтобы избежать циклических импортов:

```
core           ← базовые абстракции, ни от кого не зависит
accounts       ← зависит от core
workgroups     ← зависит от accounts
chats          ← зависит от accounts, workgroups
tasks          ← зависит от accounts, workgroups
notifications  ← зависит от accounts (через сервис, по событиям из других app)
attachments    ← зависит от accounts; FK на Task, Message, WorkGroup — nullable
dashboard      ← зависит от всех остальных, только для отображения
```

Кросс-модульные операции (например, "создание задачи → запись истории → отправка уведомлений")
оркестрируются в `apps/tasks/services.py`, который импортирует функции из
`apps/notifications/services.py`.

### Деферированные импорты (предотвращение циклических импортов)

Модуль `workgroups` создаёт рабочие группы и должен создавать для них `workgroup`-чат
(из `chats`). Но `chats` зависит от `workgroups` (FK `Chat.workgroup`). Прямой импорт
на верхнем уровне файла создал бы цикл.

**Решение**: импорт `chats.services` выполняется **внутри тела функции** (ленивый/
отложенный импорт). К моменту первого вызова функции Django уже загрузил оба приложения,
поэтому цикла нет.

Применяется в двух местах:
- `apps/workgroups/services.py::create_group` и `update_group` — `from apps.chats.services import ...`
  внутри тела функции;
- `apps/workgroups/signals.py` — обработчик сигнала импортирует `sync_workgroup_chat_members`
  внутри тела обработчика.
