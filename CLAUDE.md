# Инструкции Claude для проекта "4К"

Файл загружается в контекст автоматически при каждой сессии Claude Code в этом
проекте. Содержит правила работы и карту документации. Подробности — в файлах,
указанных ниже.

## Контекст проекта

Разрабатывается Django-веб-приложение для внутренней информационной системы компании
"4К" в рамках ВКР. Срок разработки — до 20 дней. Главная цель — рабочий веб-прототип
для демонстрации на защите.

Бизнес-контекст и MVP-сценарии — в [CLAUDE_PROJECT_CONTEXT.md](CLAUDE_PROJECT_CONTEXT.md).

- Работаешь над интерфейсом → `docs/ui.md`.

## Когда создавать новый документ в docs/

Предложи создать новый файл в `docs/`, если возникает одна из ситуаций:

1. **Новый функциональный модуль**, который заслуживает отдельного описания
   (например, отчётность, экспорт данных, интеграция с внешним сервисом).
2. **Сложный регламент**, который нельзя коротко описать в существующих документах
   (например, политика бэкапа, регламент развёртывания на проде).
3. **Раздел, который вырос внутри существующего документа** до такой степени, что
   мешает читать остальное (тогда — выносим в отдельный файл и оставляем ссылку).
4. **Новая сквозная архитектурная тема**, которая нужна нескольким модулям одновременно
   (например, кэширование, фоновые задачи, расширенная валидация форм).

В этих случаях:
1. Не создавай файл сам — сначала **сообщи мне** о необходимости и предложи название,
   назначение и краткий план содержимого.
2. После моего подтверждения — создай файл и **обнови карту документации в этом файле
   (CLAUDE.md)**, добавив строку в таблицу выше.
3. Старайся не плодить файлы для тривиальных вещей — лучше расширить существующий
   документ.

## Режим работы

Не начинай писать код сразу, если задача относится к архитектуре, моделям, правам
доступа или новому модулю. Сначала предложи план изменений и дождись подтверждения.

Если я прошу пример — не создавай файлы внутри проекта. Дай ответ в окне Claude Code
(в чате расширения VS Code или в терминале).

### Итерационный цикл

1. анализ текущего состояния;
2. план изменений с объяснением, как изменится поведение приложения;
3. список файлов, которые будут затронуты;
4. реализация после подтверждения;
5. объяснение, что изменено;
6. команды для проверки результата.

### После изменений

1. перечислить изменённые файлы;
2. объяснить изменения;
3. указать команды для проверки;
4. указать возможные риски;
5. предложить следующий шаг.

Не смешивать много модулей в одной итерации.

## Стек (зафиксирован)

Подробности и обоснование — в [docs/architecture.md](docs/architecture.md).

- Python 3.12+, Django 5.x, PostgreSQL 16;
- Gunicorn (WSGI), Docker Compose с двумя контейнерами `web` и `db`;
- файлы — через Docker volume, подключённый к web-контейнеру;
- frontend: Django Templates + Bootstrap 5 + vanilla JavaScript (без TypeScript);
- чат: polling (5 секунд), не WebSocket.

**Запрещено без отдельного обоснования и моего подтверждения**: React, Vue,
отдельный SPA frontend, Django REST Framework, WebSocket, Django Channels, Redis,
Celery, микросервисы, Kubernetes, Kafka, ASGI-сервер, дополнительные контейнеры.

Если предлагается технология сверх стека — сначала объяснить:
- зачем она нужна;
- какую проблему решает;
- насколько усложняет проект;
- можно ли обойтись без неё в рамках прототипа.

## Качество

Прототип — но проектируется как основа корпоративной ИС:
- понятные модели;
- логичные связи;
- ролевой доступ;
- проверка прав в service layer (не только в шаблонах);
- история изменений задач;
- базовая безопасность (CSRF, проверка прав на скачивание файлов, белый список
  расширений);
- читаемая структура проекта.

Если видишь риск архитектурной ошибки — остановись и сначала объясни проблему.

## Git workflow

Проект ведётся через Git. **Не выполнять Git-команды без явного подтверждения или
конкретной просьбы.** Никогда без подтверждения: `push --force`, `reset --hard`,
удаление веток, `--no-verify`.

После завершения логически цельного этапа разработки напоминай, что стоит сделать
commit. Предложение должно содержать:
- краткое объяснение, почему сейчас уместен commit;
- сообщение commit (commit message);
- команды для проверки состояния репозитория;
- команды для создания commit.

Пример формата:

```
Точка для commit:
- Причина: реализована основа авторизации пользователей и ролевого доступа.
- Сообщение commit: `Add user authentication and role-based access foundation`
```

Допускаются многострочные сообщения для уточнения деталей.

## Действия, требующие явного подтверждения

- любые `git push`, особенно `--force`;
- удаление файлов / директорий;
- удаление volumes Docker (`docker volume rm`, `docker compose down -v`);
- откат миграций Django (`migrate <app> <prev>`);
- установка новых пакетов / технологий вне согласованного стека;
- изменение CLAUDE.md, CLAUDE_PROJECT_CONTEXT.md и файлов в docs/ без явной просьбы.

## Стиль объяснений

Отвечай **по-русски**. Я junior-разработчик, у меня часто возникают пробелы в общей
архитектуре: я могу не знать, что именно я не знаю.

При первом упоминании в ответе кратко расшифровывай технические термины: ASGI, WSGI,
WebSocket, polling, AJAX, ORM, middleware, migration, volume, bind mount, static
files, media files, queryset, permission, decorator, service layer, signal,
fixture и т.п.

### Формат объяснения сложного

1. сначала простое объяснение;
2. затем технический смысл;
3. затем зачем это нужно именно в этом проекте и как изменится поведение приложения.

Не перегружай ответ теорией, но не оставляй термины без пояснения.

## Финальная таблица в каждом ответе

В конце каждого содержательного ответа (с действиями, решениями, анализом, ревью,
изменениями файлов) добавляй краткую агрегационную таблицу. Если ответ короткий
и не содержит действий или решений — таблица не нужна.

Таблица не дублирует содержимое ответа: только сжатый чек-лист.

Выбирай форму по характеру ответа:

| Случай | Колонки таблицы |
|--------|-----------------|
| Проектирование / обсуждение будущих решений | `№ пункта \| Действие или решение \| Где планируется \| Что требуется от меня` |
| Выполненная разработка (изменены файлы) | `Файл \| Что изменено \| Зачем \| Как проверить` |
| Ревью кода / архитектуры / документации | `№ пункта \| Найдено \| Важность \| Рекомендация \| Править сейчас?` |
| Исправление ошибки / отладка | `Проблема \| Причина \| Файл / место \| Исправление \| Проверка` |
| Документация (правки в .md) | `Документ \| Раздел \| Что изменено \| Зачем` |
| Планирование этапа разработки | `Шаг \| Задача \| Результат \| Зависимости \| Приоритет` |
| Завершён крупный этап | `Блок \| Статус \| Что готово \| Что осталось \| Команды проверки` |

Важность в ревью: `низкая` / `средняя` / `высокая` / `критичная`.

Правила:
- Если от пользователя ничего не требуется — пиши «Ничего, можно продолжать».
- Если требуется решение — формулируй конкретно: «Подтвердить вариант A/B»,
  «Дать имя модели», «Разрешить изменить файл».
- Если планируется изменить файлы — сначала план + таблица, код только после
  явного подтверждения (исключение: пользователь явно попросил сразу применить).
